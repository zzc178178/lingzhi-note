## Context

ai-note-2.0 技能目前在 Claude Code 环境中运行，通过模板和脚本生成精美的双页签 HTML 笔记。用户希望将这个能力普惠化，让更多用户（尤其是移动端用户）能够便捷地使用。我们决定基于 Taro 框架构建跨端应用，同时支持微信小程序、支付宝小程序和 H5 网页。

### 现有技术基础

- **ai-note-2.0 技能**：成熟的笔记生成逻辑和视觉设计系统
- **模板系统**：`template.html` 包含完整的 CSS 样式和 HTML 结构
- **构建脚本**：`scripts/build.py` 处理 JSON 到 HTML 的转换

### 技术约束

- 小程序包体积限制（微信 2MB，支付宝 3MB）
- 各平台 API 差异需要抽象
- AI 生成需要后端服务支持（客户端无法直接调用大模型）

## Goals / Non-Goals

**Goals:**

- 构建基于 Taro 3.x 的跨端应用框架
- 实现双页签笔记的核心 UI 组件
- 支持 6 种可视化布局模式（Mermaid 流程图、三列卡片、纵向步骤、CSS Flow、标签网格、CSS Tree）
- 提供笔记管理功能（生成、查看、分享、导出）
- 覆盖微信小程序、支付宝小程序、H5 三大平台

**Non-Goals:**

- 不实现用户登录和云端同步（v1.0 仅支持本地存储）
- 不实现复杂的富文本编辑器（随手记仅支持基础格式化）
- 不支持支付宝特有的能力（如蚂蚁森林等）

## Decisions

### 1. 跨端框架选择：Taro 3.x vs uni-app

**选择 Taro 3.x**

- **理由**：React 生态更成熟，组件库丰富
- **优势**：与现有 ai-note-2.0 的 React 技能体系一致
- **替代方案**：uni-app 使用 Vue，上手成本高

### 2. AI 生成方案：云函数 + API 调用

**架构**：小程序 → 云函数 → AI API

- **理由**：客户端无法安全存储 API Key
- **方案**：
  - 微信小程序：使用云开发云函数
  - 支付宝小程序：使用云函数或后端服务
  - H5：调用后端 API
- **数据流**：用户输入 → 云函数 → AI API → 解析结果 → 渲染笔记

### 3. 样式方案：CSS Modules + 模板继承

**策略**：

- 从 `template.html` 提取核心 CSS 为公共样式文件
- 使用 CSS Modules 管理组件样式
- 抽取占位符替换逻辑为通用函数
- **替代方案**：纯内联样式 → 优点是简单，缺点是难以维护

### 4. Mermaid 图表渲染：CDN + 本地降级

**策略**：

- H5：加载 `mermaid@11` CDN
- 小程序：使用本地化 Mermaid.js（编译到产物）
- **降级方案**：解析失败时显示纯文本描述
- **风险**：小程序包体积会增加约 100KB

### 5. 数据存储：localStorage + JSON 文件

**策略**：

- 使用 `localStorage` 存储笔记列表和元数据
- 每个笔记保存为独立 JSON 文件
- **H5**：使用 FileSystem Access API 提供导出功能
- **小程序**：使用 `wx.getFileSystemManager()` 写入本地文件

### 6. 页面路由设计

```
pages/
├── index/              # 首页 - 创建笔记入口
├── create/             # 创建笔记 - 输入 URL 或文本
├── note/[id]/         # 笔记详情 - 双页签展示
├── notes/              # 笔记列表 - 管理已生成笔记
└── settings/           # 设置页 - 主题、导出设置
```

## Risks / Trade-offs

### [Risk] AI API 成本控制

- **问题**：AI 生成笔记可能产生较高 API 调用成本
- **缓解**：
  - 实现缓存机制（相同 URL 避免重复调用）
  - 用户配额机制（免费次数限制）
  - 后端计费系统（预留扩展接口）

### [Risk] 小程序包体积超标

- **问题**：包含 Mermaid.js 后包体积可能超过 2MB
- **缓解**：
  - 使用 Taro 的 tree-shaking 优化
  - 按需加载 Mermaid（仅图解页签使用时加载）
  - 考虑使用轻量级图表替代方案（如 CSS + SVG）

### [Risk] Mermaid 渲染失败

- **问题**：Mermaid 语法错误或 CDN 加载失败
- **缓解**：
  - 继承 ai-note-2.0 的校验脚本
  - 提供降级展示（显示 Mermaid 源码文本）
  - 用户可切换到文字页签查看完整内容

### [Risk] 跨平台 UI 一致性

- **问题**：不同小程序的组件表现有差异
- **缓解**：
  - 使用 Taro 的跨端组件库（如 NutUI）
  - 核心 UI 使用自定义组件而非平台原生组件
  - 统一设计系统，确保视觉一致性

### [Trade-off] 本地存储 vs 云端同步

- **当前选择**：纯本地存储（v1.0）
- **优点**：实现简单、用户无门槛、数据隐私
- **缺点**：换设备无法同步、多端体验割裂
- **未来规划**：v2.0 提供用户登录和云端同步

## Migration Plan

### Phase 1: MVP 发布（H5 优先）

1. 搭建 Taro 项目框架
2. 实现核心 UI 组件（双页签、6 种布局）
3. 集成后端 API（Mock 数据测试）
4. H5 版本上线测试

### Phase 2: 小程序适配

1. 适配微信小程序
2. 适配支付宝小程序
3. 测试各平台差异
4. 申请小程序上线

### Phase 3: 功能完善

1. 实现笔记管理功能
2. 添加分享和导出能力
3. 优化性能和体积
4. 用户反馈迭代

## Open Questions

1. **后端服务选型**：使用现有的 AI 服务还是自建？需要评估成本和稳定性
2. **用户体系**：是否需要用户登录？影响数据同步和配额管理策略
3. **内容审核**：AI 生成的内容是否需要审核机制？
4. **商业化模式**：免费版限制 vs 付费版功能
5. **Mermaid 本地化**：小程序中 Mermaid.js 的最佳集成方式？需要测试包体积影响
